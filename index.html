Before Anything Else
GET YOUR MACHINE READY

Zero assumptions. If you've never opened a terminal, start here. This gets you from a brand-new computer to running Claude Code in about 20 minutes.

Step 1
Open the Terminal
The terminal is a text window where you type commands to control your computer. Think of it as talking directly to your Mac without clicking anything.
How to open it
Press ⌘ Command + Space to open Spotlight, type Terminal, hit Enter. You'll see a black or white window with a blinking cursor. That's it — you're in.
💡 Pro tip

Drag Terminal to your Dock now. You'll use it constantly. Alternatively, download iTerm2 — it's a better terminal app, free, and what most developers use.

Step 2
Install Homebrew (Mac's Package Manager)
Homebrew lets you install developer tools with a single command. You need this to install Node.js cleanly. Copy the command below, paste it into your terminal, and press Enter.
# Paste this entire line into your terminal and hit Enter /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" # It will ask for your Mac password — type it (nothing will show) and hit Enter # Takes 2-5 minutes. When done, verify with: brew --version
ℹ️ What's happening

The terminal will show a lot of text scrolling by — that's normal. It's downloading and installing Homebrew. When you see a $ prompt again, it's done.

Step 3
Install Node.js
Node.js lets your computer run JavaScript outside a browser. Claude Code requires it. With Homebrew installed, this is one command.
# Install Node.js via Homebrew brew install node # Verify — should print something like "v20.x.x" node --version # Also verify npm installed alongside it npm --version
Step 4
Install Claude Code
Now install Claude Code — Anthropic's CLI tool that lets you build with Claude directly from your terminal.
# Install Claude Code globally via npm npm install -g @anthropic-ai/claude-code # Verify it installed claude --version # Launch it (always run this from inside your project folder) claude
⚡ First Launch

The first time you run claude, it will ask you to log in with your Anthropic account. You need a Pro or Max plan at claude.ai to use Claude Code.

Step 5
Install VS Code (Your Code Editor)
You need a place to view and manually edit files. VS Code is free, universally used, and works great alongside Claude Code.
Download and install
Go to code.visualstudio.com → Download for Mac → Open the .zip → Drag VS Code to your Applications folder.
# Add the 'code' command to your terminal: # Open VS Code → press ⌘ Shift P → type "shell command" # → click "Install 'code' command in PATH" # Now open any folder in VS Code directly from terminal: code .
Step 6
Install Git and Set Up GitHub
Git tracks every change to your code. GitHub stores it in the cloud. Vercel deploys from GitHub. This chain is your entire delivery pipeline.
# Install latest Git via Homebrew brew install git # Tell Git who you are (use your GitHub email) git config --global user.name "Your Name" git config --global user.email "[email protected]" # Verify git --version
GitHub account
Go to github.com → Sign up (free) → Create a new private repository for your project.
Step 7
Create Your First Project and Start Building
Everything is installed. Here's the exact sequence to go from zero to Claude Code running in a real project.

⚡ The Rule of Thumb

PROMPTING LIKE YOU MEAN IT

Bad prompts waste hours. Great prompts ship products. Here's how to talk to Claude Code like a founder, not a confused user.

Plain English
You're the CEO. Claude is your engineer. Prompts are your briefs.
A prompt is just the message you type to Claude. But like any good hire, Claude performs best when you give it full context — not just "make a login page" but who it's for, what stack you're using, what it should look like, and what not to break.

Think of every prompt as a one-paragraph brief to a contractor. The more specific you are upfront, the less you go back-and-forth. The less you go back-and-forth, the faster you ship.
💬
The analogy: Imagine hiring a brilliant but literal-minded freelancer on Upwork. If you say "design my website," you'll get something generic. If you say "design my SaaS dashboard — dark theme, data-dense, inspired by Linear, using these colors" — you get something useful on the first try. Same energy.
⚡ The Core Rule

Claude Code is not a search engine. It's a senior engineer who needs a brief. The more context, constraints, and clarity you give upfront, the less back-and-forth you waste. Think of it like onboarding a contractor in a single message.

✗ Weak Prompt
What you saidVAGUE
"Make me a landing page for my startup."

You'll get a generic template. No personality, wrong assumptions, 10 revisions ahead.

✓ Strong Prompt
What you should sayCLEAR
"Build a landing page for Launchpad, a project management tool for remote teams. Dark background, minimal, one CTA: 'Start Free Trial'. Hero headline: 'Ship faster, together.' Use Next.js and Tailwind."
🗂️
Give it a CLAUDE.md file
What it isA plain text file named CLAUDE.md that lives in your project folder. Claude reads it automatically at the start of every session — like handing a new hire your company handbook on day one. No hunting through old conversations. No re-explaining your stack.

Write: your tech choices, your design rules, what Claude should never touch. One setup, infinite sessions.
🎯
One Task Per Prompt
Don't chain 5 things in one message. "Build the auth flow. Then the dashboard. Then..." — pick one thing, get it done, move on. Claude loses context on sprawling asks.
🧠
Tell Claude what you want, not how to build it
Say "I want a button that saves the user's email to Supabase" — not "write a POST request to..." Let Claude figure out the implementation.
🔁
Use /compact to clear Claude's memory
What it isClaude has a memory limit — after enough back-and-forth, it starts "forgetting" earlier context and making mistakes. Type /compact and Claude summarizes everything important into a fresh start, without losing your work.

Think of it like saving your game before a boss fight. Do it every 20–30 messages, or whenever Claude starts giving weird answers.
📸
Show Claude, don't describe
Why it worksClaude can see images. Instead of typing "make it look clean and modern like Stripe," drag a screenshot of Stripe's dashboard directly into the chat. Visual reference beats verbal description every time — Claude matches layout, spacing, and color choices from a real image far better than from adjectives.

Drag a screenshot of a UI you love directly into Claude Code. Say "make mine look like this". Visual reference beats 500 words of description every time.
⛔
Constraints are your friend
Tell Claude what you don't want: "Don't add a sidebar. Don't use TypeScript. Don't install new packages." Guardrails prevent scope creep.
🔥 Power Prompts to Steal

"Think step by step before writing any code." Adds a reasoning pass.

"Explain what you're about to do before doing it." Catches misalignment early.

"Don't change anything I didn't ask you to change." Stops scope creep.

"If you're unsure about my intent, ask before building." Saves full rebuilds.

DEFINING YOUR FEATURES

Before you build, lock down what you're building. Claude will happily build everything you ask — including stuff you don't need.

Plain English
A feature is anything your user can do. Start with the 3 that matter most.
Before you write a single prompt, you need a list of what your product actually does. Claude will happily build anything you ask — including things you don't need yet. A feature list keeps you from building a dashboard before you have users.

The format that works best: "As a [type of user], I want to [do something] so that [outcome]." This forces you to think from your user's perspective, not just what's cool to build.
🏗️
The analogy: You wouldn't hire an architect and say "build me a house." You'd give them a brief: 3 bedrooms, open kitchen, no formal dining room. Feature stories are your product brief. Without them, Claude builds a house with 6 bathrooms and no kitchen.
📐 Do This First

Write your feature list as user stories before touching Claude Code. "As a [user], I want to [action] so that [outcome]." It forces clarity. Claude then translates it into code, not assumptions.

🗺️
Start with a scope doc
What it isA simple list of every feature your product needs — written in plain language, not code. Before you touch Claude Code, paste this list into a regular Claude chat and ask: "help me prioritize this for an MVP."

Claude will cut the list down to what actually matters for a first user, and put the rest in a "later" column. Then you build in that exact order. No scope creep. No 3-month builds.
🚦
MVP vs V2 vs Someday — tag everything
What MVP meansMinimum Viable Product — the smallest version of your idea that a real person would actually use. Not everything. Not polished. Just the one core thing that works.

Tag every feature: MVP = must have for first user. V2 = adds after first 10 users. Someday = only if it becomes a real business. Ship MVP only.
💬
Ask Claude to challenge you
Prompt: "Here's my feature list. Play devil's advocate — what am I building that users probably don't care about?" Brutal but useful.
🔗
Map dependencies first
Some features depend on others. Auth before dashboard. Database schema before forms. Tell Claude: "What's the correct build order for these features?"
📋
Write a PRD, paste it in
Even a one-pager. Product, users, problem, success metrics. Claude builds with more precision when it understands why, not just what.
🎪
Fake it first
Build a static mockup with hardcoded data before wiring up the backend. Validate the UX is right. Then make it real. Saves massive rework.
Write user stories for every feature before prompting"As a [user], I want to [X] so that [Y]"
Tag features: MVP / V2 / SomedayOnly build MVP until you have real users
Ask Claude to identify the build orderAvoid blocked dependencies mid-sprint
Build a static mockup before wiring the backendValidate UX first, then make it real
Have Claude review your feature list for bloat"What am I overbuilding for my stage?"

Design & Feel
UI / UX THAT DOESN'T SUCK

Claude can make things beautiful. But it defaults to generic. Push it. Every effect below is live — interact with it, then steal the prompt and code.

Plain English
UI is what it looks like. UX is how it feels to use. Both are learnable.
UI (User Interface) = the visual layer. Buttons, colors, fonts, layout. UX (User Experience) = how smooth and intuitive it is to actually use. You've experienced bad UX every time an app made you feel dumb. Good UX is invisible — it just works.

Claude can produce both — but only if you push it with specific references and explicit animation requests. Left to its defaults, it builds functional but forgettable. The prompts in this section are how you close that gap.
🎬
The analogy: UI is the set design and costumes of a film. UX is the pacing, the score, the editing — how the whole thing flows. A great movie needs both. Your product does too. "Clean and modern" is not a direction. "Like Airbnb's checkout flow, minimal, one action per screen" is.
🎨 The Golden Rule

Reference real products you love. "Make it feel like Linear's onboarding." "Match the density of Notion." "Transition like the iPhone's springboard." Shared vocabulary beats abstract adjectives.

Specify animations explicitly in your promptsClaude won't add motion unless you ask
Add hover states to every interactive elementScale, color shift, shadow — pick one, be consistent
Implement scroll-triggered revealsIntersectionObserver + CSS transitions = perceived polish
Add haptic feedback to key mobile actionsnavigator.vibrate() for web, expo-haptics for React Native
Build loading, success, and error states for every async actionSkeleton loaders > spinners > blank screens
Declare mobile-first at the start of every UI promptOr Claude will assume desktop layout
Reference real apps for aesthetic direction"Like Linear" / "Like Notion" beats "clean and modern"
Test on a real device before calling it doneBrowser DevTools lies. Your thumb doesn't.

04
Data & Logic
BACKEND WITHOUT THE PAIN

You don't need to understand databases deeply. You need to know what to connect and how to ask Claude to wire it up properly.

Plain English
The backend is everything your users never see — but always depend on.
Your frontend is the interface — what people click and see. Your backend is the engine: the database that stores data, the logic that processes it, the auth system that knows who's logged in. If the frontend is the restaurant floor, the backend is the kitchen.

You don't need to cook — you need to pick the right kitchen setup and tell Claude exactly how to wire it. This section gives you the opinionated stack that works, why each piece exists, and the prompts that set it up correctly.
🍽️
The analogy: When you use an app to order food, you see a clean screen. Behind it: a database of restaurants, a payments processor, a user account system, and servers running logic. You built all of that when you prompt Claude to set up Supabase + auth + API routes. One afternoon. No CS degree required.
Database
Where all your data lives. Users, orders, messages — everything stored so it persists between sessions. Think: a very powerful spreadsheet in the cloud.
Auth
Short for Authentication — the system that handles sign-up, login, and "is this user allowed to see this?" So you never build a login screen from scratch.
API
The bridge between your frontend (what users see) and your backend (where data lives). When you click "save," an API call carries your data to the database.
Environment Variables
Secret values (API keys, passwords) stored outside your code so they never get accidentally shared. Saved in a file called .env that Git ignores.

🗄️ Supabase Prompt Template

"Set up a Supabase table called 'subscribers' with columns: id (uuid, primary key), email (text, unique), created_at (timestamp), source (text). Enable Row Level Security. Users can only INSERT, not read others' data." This level of specificity gets it right first time.

🔐
Row Level Security (RLS)
What it isImagine a shared spreadsheet where each user can only see their own rows — even though all the data lives in one place. That's Row Level Security. Without it, a logged-in user could theoretically request anyone else's data.

Supabase makes this easy. Always prompt: "Enable RLS on every table. Users can only read and write their own rows where user_id = auth.uid()." One prompt. Done.
🌱
Seed data — fake data for testing
What it isWhen you're building, your database is empty. That makes it impossible to see how your UI actually looks with real content. Seed data is a set of fake-but-realistic records you pre-load so you can design and test against something real.

Ask Claude to "create a seed.sql file with 10 sample records". Develop against real-feeling data. Catch edge cases before real users do.
📡
Real-time updates — live data without refreshing
Real-time subscriptions are powerful but can create complexity. Only use it where live updates are core to the UX — chat, collaborative editing, live dashboards.
🧪
Test locally before touching your live database
Run supabase start locally. Never build against production while prototyping. Claude can scaffold the local setup in one prompt.

GETTING IT LIVE

Deploy early and often. A product in production teaches you more in a week than months of local dev.

Plain English

GITHUB FOR NON-ENGINEERS

GitHub is your save system. Commit often, branch for features, never push directly to main. Claude can do most of this for you.

Plain English
GitHub is Google Docs version history for your code — but you control when to save.
When you write code, things break. Git is a system that tracks every change so you can always go back. GitHub is the cloud storage that holds those saves online and lets you share them with teammates or deploy to hosting platforms.

You'll use three concepts constantly: commits (a named save), branches (a parallel version where you try things without breaking the live app), and push (syncing your local saves to GitHub). Claude can do all three for you — you just need to understand why.
📖
The analogy: Imagine writing a book. Git is your version history — "Draft 1," "After editor notes," "Final final ACTUAL final." A branch is opening a copy to try a new ending without touching the original. A commit is hitting Save with a note to your future self about what changed.
💾 The Mental Model

Think of GitHub like Google Docs version history, except you control when to save and what to name each save. Commits = saves. Branches = separate drafts. Main = the one that's live.

🌿
Branch for every feature
What a branch isA parallel copy of your code where you can build and experiment without affecting the live version. Think: opening a duplicate Google Doc to try edits without changing the original. Main is the version that's live on the internet. Everything else is a branch.

Never build directly on main. Ask Claude: "Create a new branch for this feature and commit when done."
📝
Write real commit messages
What a commit isA named snapshot of your code at a moment in time. Like hitting "Save As" with a description of what changed. If something breaks two weeks from now, you'll scroll through commits to find where it went wrong — so the message matters.

Not "update" or "fix". Say "add email capture form with Supabase integration". Future you will thank present you. Claude will write them if you ask.
🔍
.gitignore — tell Git what NOT to save
Prompt: "Create a .gitignore for Next.js with Supabase. Exclude .env, node_modules, .next, and any local config." Do this before the first push. Non-negotiable.
👁️
Private repo, always
Unless you're deliberately open-sourcing, keep it private. Your .env history, your business logic, your competitive advantage — none of it should be public by default.

SECURITY BASICS (NON-OPTIONAL)

You don't need to be a security engineer. You need to not make the obvious mistakes. These 10 things cover 90% of what matters at your stage.

Plain English
Security is locking the doors before you invite the public in.
You don't need to understand cryptography. You need to know where the obvious unlocked doors are and close them before you launch. The most common startup breaches aren't sophisticated hacks — they're API keys accidentally posted to GitHub, or databases with no access rules.

The checklist in this section covers the 10 things that matter at your stage. Each one has a Claude prompt you can run right now. Think of it as a pre-flight check, not a PhD program.
🔐
The analogy: Opening a coffee shop? You lock the register, don't leave the safe open, and check IDs for alcohol. You don't hire a CIA security team. Same principle here — basic hygiene eliminates 90% of the risk. The other 10% you'll deal with when you're much bigger.
⚠️ Reality Check

You will get attacked the moment you're indexed by Google. Bots don't care that you're an early startup. The basics below take 2-4 hours to implement. The breach takes 2 minutes to exploit.

🔑
Never put secret keys in your code
What a secret key isEvery service you connect to (Stripe, Supabase, email) gives you a password-like key to identify your app. If that key ends up in your code and you push to GitHub, anyone can find it and use your account. This happens constantly.

Instead, put all keys in a file called .env.local — Git ignores this file automatically. Claude prompt: "Audit this file for any hardcoded credentials."
🛡️
Row Level Security — each user sees only their data
Why it mattersWithout RLS, any logged-in user could request every row in your database — every other user's data, orders, messages, everything. With RLS on, the database enforces "you can only see rows that belong to you" at the database level itself.

Supabase's RLS means users can only see their own data. Without it, anyone with your anon key can read your entire database. Always on. No exceptions.
📧
Verify email before granting access
Enable email confirmation in Supabase Auth. One setting. Prevents fake accounts, spam, and some abuse vectors. Ask Claude to add a "check email confirmed before showing app" gate.
🚫
Rate limiting — stop bots from hammering your app
Prompt: "Add rate limiting to this API route — max 10 requests per minute per IP." Use Upstash Redis for this (free tier). Prevents abuse and cost explosion.
🌐
HTTPS everywhere (free on Vercel)
Vercel handles SSL automatically. Just make sure you're not loading any HTTP resources (images, scripts) on an HTTPS page. Ask Claude to "audit for mixed content issues."
🧹
Sanitize inputs — never trust what users type
Anything a user types that ends up in your database or gets rendered on screen needs sanitization. Prompt: "Add input sanitization and validation to every form in this file."
🔐
Use Supabase Auth, not DIY
Building your own password system is one of the most dangerous things a non-security-engineer can do. Supabase Auth is battle-tested. Just use it.
👁️
Audit your public API surface
Prompt: "List every API route in this project. For each one, tell me if it requires auth and what data it exposes." Close anything that shouldn't be open.

Plain English
You shipped. Now make it smarter, faster, and more data-driven.
You have a working product. This section is about the habits that separate founders who iterate fast from those who disappear into building mode. Analytics tells you what's actually being used. Feature flags let you test without breaking things. Automation saves you from repetitive work.

None of this requires a full engineering team. With Claude, each of these is an afternoon of prompting, not weeks of work. Stack these habits early and you'll compound speed over every sprint.
🚦
The analogy: You opened the restaurant. Now you install a POS system to track which dishes sell, put one new special on the menu without changing everything, and automate the weekly inventory order. Same energy — small systems that multiply your output without multiplying your hours.
📊
Analytics — see what users actually do
What it isA tool that records every click, page view, and action in your app — so you can see which features get used and which get ignored. Without it, you're building blind, guessing what matters.

Add PostHog or Plausible before your first user. Free tiers are generous. Claude can wire up event tracking in one prompt: "Add PostHog tracking to every button click and page view."
🧪
Feature flags — ship to some users, not all
What it isA switch that lets you turn a feature on for specific users without deploying new code. You can test a new onboarding flow with 10% of users, see if it works, then roll it out to everyone — or quietly turn it off if it doesn't.

Use PostHog's feature flags or LaunchDarkly to ship features to specific users first. "Enable this for user X only" — test without touching code.
🤖
AI features are easy now
Want to add AI to your product? "Add an AI chat feature using the Anthropic SDK that answers questions about [topic]." Shipping an AI feature is now a 2-hour task.
🔔
Webhooks — let other services notify you automatically
What it isInstead of your app constantly asking "did anything change?" (polling), a webhook means Stripe or Supabase will tap you on the shoulder the moment something happens — payment succeeded, user signed up, database row changed.

It's faster, cheaper, and more reliable. Stripe webhooks tell you when payments succeed/fail. Supabase webhooks fire on DB changes. Don't poll — listen. Claude can scaffold a webhook handler in minutes.
📝
Document as you build
End every session: "Summarize what we built today and update the README." Future-you and your co-founder will be grateful. It also forces clarity.
🔄
Automate with Claude, not code
Repetitive tasks? "Write a script that does X every time Y happens." Claude can automate your own workflow. You should never do the same tedious thing twice.

Component libraries are your friend
Don't build every UI element from scratch. Use shadcn/ui (free, beautiful, customizable). Prompt: "Use shadcn components for this UI. Install whatever's needed."
🚀
Ship, then optimize
Premature optimization is the enemy of shipping. Get it working, get it in front of users, then — and only then — make it fast. Claude can help you profile performance when you're ready.

HOW TO LAUNCH

Most founders overbuild and underlaunce. This is the playbook for going from "it works on my laptop" to real people using your product — without waiting until it's perfect.

Plain English
Launching is not a moment. It's a habit you start earlier than you think.
There is no "ready." There's working enough to learn from. The founders who move fastest aren't the ones who build the most — they're the ones who put something in front of real people earliest and let feedback drive the next sprint.

This section covers the full launch sequence: from your first private beta link to your first public post to your first paying customer. Each step is a one-day job with Claude. You don't need a launch team. You need a checklist and a deadline.
🎯
The analogy: A restaurant doesn't wait until every dish is perfect to open. They do a soft launch — friends and family, limited menu, real feedback. Then they iterate. Your product launch works exactly the same way. Ship the limited menu first.

Your app is live on Vercel. Now send it to 10 people you trust: co-founders, advisors, close friends in your target market. No announcement. No Product Hunt. Just a Slack message or a text that says "I built something — can you try it and tell me what breaks?"
